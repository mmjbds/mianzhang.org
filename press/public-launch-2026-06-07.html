Public Release - 2026-06-07

Without schema, paper evidence drifts.

Artifact: Upload Matrix Registry Validator v0. If the 16-paper upload matrix has no registry schema, PDF, manifest, claim boundary, hash, and DOI/status records slowly drift across versions.

Validator

The matrix needs records that can fail.

The current public record is tied to W0, P23, P28, P29, P30, P31, P32-P40, and F1/P8. A validator does not make those papers stronger. It makes the public upload matrix less ambiguous by checking whether each public row has the identifiers, routes, boundaries, freshness fields, and failure status needed for review.

Today defines a public validation shape for paper, claim, evidence, counterexample, boundary, freshness, DOI/status, and hash records. It validates public registries and upload-package fields only. It does not read protected logs, customer data, account state, or protected execution systems.

1

Required fields

paper_id, file_id, claim_id, evidence_id, boundary_id, public_route, hash_status, doi_status, and last_verified.

2

Failure modes

orphan paper_id, missing public_route, broken evidence link, hash mismatch, missing boundary, stale status, or ambiguous claim.

3

Validator report

Each row should say pass, fail, warn, needs_human_review, and the exact public field that caused the result.

4

Boundary check

The validator checks public fields; it does not demand protected operational systems, customer logs, account state, or protected execution traces.

5

Challenge route

Challenge a registry field if it causes ambiguity between paper, evidence, boundary, DOI/hash, freshness, or claim status.

6

Repair action

A valid failure should produce a public repair: add a route, downgrade a claim, mark stale, update a boundary, or request evidence.

Drift Prevention

Identifiers hold versions together.

The validator is not a trust badge. It is a way to stop version drift: a PDF changes but the manifest does not, a claim boundary moves but the claim_id stays vague, a hash is replaced but freshness is not updated, or a DOI/status field becomes ambiguous.

A passing row should mean only this: the public registry has enough structure for a reader, reviewer, or tool to trace the public claim to public evidence and a public challenge route.

Challenge

Attack one registry field.

The useful critique for this artifact is small and sharp: name a registry field that makes a paper, evidence object, boundary, DOI/hash status, freshness state, or challenge route ambiguous.

Do not ask for non-public material. The validator must stay public-first: schemas, manifests, registries, demos, boundaries, issue routes, and negative results.

Public / Private Boundary

Validate public records, not private systems.

This validator must not become a back door into protected material. It excludes protected operational systems, customer data, account state, protected execution logs, sensitive instructions, protected orchestration, commercial schedulers, financial execution details, alpha hints, and trading advice.
